I **benchmarked** 5 data formats for LLM prompts. The results broke my **biased** assumptions.

Same record. Same data. Different formats:

| Format   | Tokens | Overhead |
|----------|--------|----------|
| CSV      | 10     | 0%       |
| TXT      | 15     | +50%     |
| Markdown | 16     | +60%     |
| JSON     | 20     | +100%    |
| XML      | 28     | +180%   |

![Token Tax by Format](/assets/images/token-tax-chart.svg)

Markdown tables beat JSON. Not a typo.

### Why Markdown Wins

BPE tokenizers collapse Markdown's whitespace into single tokens. JSON's key names—such as "city", "country", and "population"—cost a full token each, repeated across every row. At scale, that **overhead** compounds fast.

### The Real Problem: Context Drift

But the token count is the easy part.

The real problem with agentic workflows is what happens when you hit the **context limit**. Older conversations get **compressed**. If your agent is mid-refactor on a 40-file codebase and loses the context of the first 10 files, it doesn't crash; it **drifts**. It starts making decisions that contradict earlier ones, quietly.

### A Simple Memory Structure

You handle this with a simple three-file memory structure: a project **conventions** file, an auto-updated **patterns** log capped at 200 lines, and a local dev **context** file that never touches the repository.

### Stop Letting Agents Skip Approval

The other thing I stopped doing: letting agents **skip** the approval phase before execution. One unchecked run and you've got 50 generated files in the wrong directory, all internally consistent, **all wrong**.

### Rules That Held Up

A few rules that held up across every project:

- Tabular data goes in CSV, not JSON
- Instructions live in Markdown, not DOCX or PDF
- XML is almost never the right answer
- No agent executes without a human-approved strategy step

**Context** is a finite resource. Most people don't treat it that way yet.
