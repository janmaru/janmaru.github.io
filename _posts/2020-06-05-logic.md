If art mocks life then what is life mocking?

**Keith Haring** was an artist of the people. His murals appeared in subway stations, on warehouse walls, in public spaces where anyone could see them. His style was unmistakable — bold lines, vibrant colors, figures packed densely together, filling every inch of available space.

This is **horror vacui**: the fear of empty space. Every surface must be covered. Every gap must be filled.

## Horror Vacui and Null

There is a strange parallel between horror vacui and the **null reference** in object-oriented programming.

In art, the fear is of emptiness — of leaving something undefined. In code, the problem is exactly the opposite: we *allow* emptiness where it shouldn't exist.

Consider a simple C# interface:

```csharp
public interface UserFactory
{
    User GetById(int id);
}

public class User
{
    public int Id { get; set; }
}
```

The interface promises to return a **User**. It's a contract. You call `GetById`, you get a User back. Simple.

But the implementation lies:

```csharp
public User GetById(int id)
{
    return _users.FirstOrDefault(u => u.Id == id);
}
```

`FirstOrDefault` returns **null** when no match is found. The contract says User, but the reality says *maybe* User, *maybe* nothing.

Tony Hoare, who introduced the null reference in 1965, later called it his **"billion-dollar mistake."**

## Patching the Problem

C# tries to mitigate this with the **null-coalescing operator** (`??`):

```csharp
var user = GetById(42) ?? new User { Id = 0 };
```

And the **Elvis operator** (`?.`):

```csharp
var name = user?.Name ?? "Unknown";
```

These are band-aids. They acknowledge the disease without curing it. The fundamental issue remains: the type system allows null where it shouldn't.

## The Functional Solution

In **functional programming**, null doesn't exist.

Instead, you use the **Option type** — also known as Maybe. A value is either **Some(value)** or **None**. There is no in-between. There is no lying.

In **F#**, this looks like:

```fsharp
type User = { id: int }
let list_users = [{id=1}; {id=5}]

let GetById (id:int) =
    let found = query {
        for r in list_users do
        where (r.id = id)
    } |> Seq.toList

    match found with
    | [] -> None
    | x -> Some (x |> List.head)
```

When you call `GetById`, the return type is `Option<User>`. The compiler **forces** you to handle both cases:

```fsharp
match GetById 42 with
| Some user -> printfn "Found user %d" user.id
| None -> printfn "User not found"
```

You cannot forget. You cannot ignore the absence. The type system encodes the possibility of emptiness directly.

## No Logic in Null

In functional programming, there is no null. There is no ambiguity. You must confront the void explicitly, handle it deliberately, and move on.

Keith Haring filled every space because he feared emptiness. Functional programming fills every case because it respects it.

There is no logic in null. There never was.
